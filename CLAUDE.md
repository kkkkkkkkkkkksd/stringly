# CLAUDE.md — Stringly (TMS)

Контекст-файл для AI-агентов. Читай его и связанные `docs/` **перед** началом любой задачи.

## О проекте

**Stringly** — веб-сервис управления переводами (TMS): строки локализации хранятся в БД,
редактируются в таблице «ключ × язык» с разделами (namespaces), отдаются клиентским
приложениям по API-токену. Сейчас этап разработки фронта; бэка пока нет — работаем на
mock-слое (MSW), переключение на реальный API — флагом `VITE_API_MODE`.

## Стек

React 18 + TypeScript (strict) · Vite · react-router v6 · TanStack Query v5 ·
TanStack Table v8 + Virtual · Zustand · React Hook Form + Zod · Tailwind · MSW · Vitest.

## Командный режим (короткие запросы)

Пользователь даёт **короткую фразу** — ты сам находишь, что читать, и начинаешь. Не проси
переписывать длинные ТЗ. Протокол на любую команду из карты ниже:

1. Открой `docs/12-build-prompts.md`, найди соответствующий **Шаг**.
2. Прочитай docs, перечисленные в этом шаге (+ всегда: этот файл, `04`, `06`, `07`).
3. Посмотри **вперёд** по `10-roadmap`/`12`: что будет дальше — заложи это заранее.
4. Если шаг **UI** (помечен 🎨) → сперва предложи **2–3 варианта дизайна** и жди выбора
   (см. `13`). Не пиши финальный UI-код до утверждения.
5. Если требования неоднозначны → задай 1–3 коротких вопроса. Иначе — выполняй до «Готово, когда».
6. В конце кратко отчитайся: что сделано, допущения, точки расширения.

## Обязательные правила

1. **Слои и зависимости:** `app → pages → features → entities → shared`. Импорт только
   «вниз». `features` не зависят друг от друга (общее — в `entities`/`shared`).
2. **Данные только через React Query** + централизованная `qk`-фабрика ключей. Никакого
   «голого» fetch в компонентах. Реализация api за интерфейсом, mock↔real через `VITE_API_MODE`.
3. **Всё привязано к активному проекту (`pid`).** Даже пока переключатель проектов —
   заглушка, структура данных и ключи кэша это учитывают.
4. **Языки централизованы** на уровне проекта — колонки таблицы берутся из них.
5. **Роли/права** через `shared/lib/permissions` (`can`, `usePermission`, `<Can>`).
6. **TypeScript strict, без `any`.** На границе API — Zod-схемы (типы = `z.infer`).
7. **Производительность таблицы** не ломаем: виртуализация, мемоизация ячеек, точечные
   подписки, батч-сохранение (см. `08`).
8. **Заглушки оставляй заглушками:** видимый, но неактивный UI + готовая точка подключения.
9. **Общие компоненты — в `shared/ui` (баррель `@/shared/ui`).** Любой стилизованный
   элемент, который может понадобиться ещё где-то (кнопка, бейдж, инпут, модал, вкладки,
   логотип…), выноси сюда и переиспользуй. Не дублируй стили по экранам, не плоди «снежинки».
10. **Все тексты UI — в `shared/resources/i18n` (баррель `@/shared/resources/i18n`).** Ни
    одной видимой строки (заголовки, лейблы, кнопки, плейсхолдеры, сообщения ошибок/
    валидации) прямо в компонентах. Общие слова («Сохранить», «Скопировать»…) — в `common`;
    тексты экрана — в его namespace (`landing`, `auth`, …). Берём через `texts.<ns>.<key>`.
    Не путать с переводами-строками продукта (теми, что юзер редактирует в таблице).
11. **Сервисы — в `shared/services`, чистые хелперы — в `shared/core`.** Любую
    сетевую/интеграционную логику или сайд-эффект (network, analytics, storage) выноси в
    `services`; любую чистую переиспользуемую функцию (валидатор, форматтер, утилита) — в
    `core`. Не держи такую логику внутри компонентов/фич. (Аналог Services/Core в iOS.)
12. **Ассеты — в `shared/resources/assets` (баррель `@/shared/resources/assets`).** Иконки
    и изображения — именованными компонентами с человекочитаемыми именами (`TableRowsIcon`,
    `BrandMarkIcon`…). Никаких «сырых» SVG-`path`-строк прямо в JSX. `resources` = `assets`
    (иконки/картинки) + `i18n` (тексты).

## Дизайн-first (обязательно для UI)

Перед версткой любого экрана/флоу — **сначала предложи 2–3 варианта дизайна** (мокапы +
плюсы/минусы), дождись выбора пользователя, потом реализуй. Подробности — `docs/13`.

**Визуал зафиксирован** в `docs/14-design-system.md` (направление Indigo & Slate, шрифт
Inter, темы light+dark). Цвета/отступы/радиусы бери **только** оттуда (CSS-переменные/
Tailwind-токены), не вводи сырые hex в компонентах. Новый токен — добавляй в `14`, не плоди разовые.

## MVP: что заглушка / отложено

- **Сессия:** один бессрочный токен, без refresh.
- **Переключатель проектов** и **«+ Проект»** — заглушки (один проект, создаётся в онбординге).
- **Подключения / API-токены** — вкладка-заглушка (реализуем позже).
- **Поиск и фильтры** в таблице — UI неактивен (следующая итерация).
- **Дашборд, импорт/экспорт, история, автоперевод** — v2/later.

## Когда задавать вопросы

- Требования экрана/задачи **неоднозначны** → задай 1–3 коротких вопроса **до** написания
  кода, не угадывай.
- Решение влияет на **архитектуру или будущие шаги** → опиши варианты и допущения, спроси.
- UI-задача → сначала **дизайн-варианты** (это форма вопроса «что выбираем»).
- Если docs противоречат запросу → не молчи, уточни, что приоритетнее.

## Стиль работы

Чётко и лаконично. Делай ровно то, что в текущем шаге; не расширяй scope без спроса.
Перед «вперёд» смотри в `10-roadmap` и `12-build-prompts` — что будет дальше, чтобы не
переделывать. Краткий комментарий к нетривиальным решениям и допущениям.

## После каждой задачи: проверка и переиспользование (обязательно)

Перед тем как сказать «готово», прогони этот ритуал (как ревью в команде):

1. **Сверься с тем, что сделано раньше.** Загляни в `src/shared/ui`, `shared/core`,
   `shared/services`, соседние фичи и экраны: нет ли уже готового компонента/хелпера/сервиса
   под твою задачу. Если есть — используй его.
2. **Найди повторы и не-UI-логику.** Стилизованный элемент → в `shared/ui`. Сетевая/
   интеграционная логика или сайд-эффект → в `shared/services`. Чистый хелпер/валидатор/
   форматтер → в `shared/core`. Не оставляй копипаст и не держи такую логику в компонентах.
3. **Обнови баррель** `@/shared/ui` (index) и, если ввёл новый паттерн/токен, отметь это в
   `docs/14-design-system.md`.
4. **Прогон проверок:** `npm run typecheck`, `npm run lint`, `npm test`, `npm run build` — зелёные.

> Это стандартный для веба подход — единая библиотека UI-компонентов (дизайн-система),
> аналог переиспользуемых `View` в iOS. Цель — не настраивать стили заново на каждом экране.
