# 05. Технологический стек

Стек выбран под главные требования: **производительность таблицы**, **масштабируемость**,
**понятность**. Для каждой технологии — зачем она и почему именно она.

| Слой | Выбор | Зачем / почему именно это |
|------|-------|---------------------------|
| Язык | **TypeScript (strict)** | Типобезопасность по всему пути данных; критично при большом числе ключей/языков |
| Сборка | **Vite** | Мгновенный dev-старт, простая конфигурация, нативный ESM |
| UI | **React 18** | Требование заказчика; зрелая экосистема |
| Роутинг | **react-router v6** | Требование; nested routes идеально ложатся на `/app/settings/*` и гарды |
| Серверные данные | **TanStack Query (React Query) v5** | Требование; кэш, инвалидация, ретраи, пагинация, оптимистичные апдейты «из коробки» |
| Таблица | **TanStack Table v8** | Headless — полный контроль над рендером; колонки/сортировка/фильтры как данные |
| Виртуализация | **TanStack Virtual** | Рендер только видимых строк → плавность на 10k+ строк (см. ниже) |
| Клиентское состояние | **Zustand** | Минималистичный, без boilerplate Redux; идеален для auth/проекта/буфера правок |
| Формы | **React Hook Form** | Неконтролируемые инпуты = мало ререндеров; быстрые формы |
| Валидация | **Zod** | Одна схема = типы + рантайм-валидация форм И ответов API |
| HTTP | **fetch + тонкая обёртка** (или axios) | Интерсепторы для токенов/ошибок; без лишних зависимостей |
| Мок сети | **MSW** | Перехват на уровне сети → фронт работает как с реальным API без бэка |
| Стили | **Tailwind CSS** (или CSS-модули) | Быстрая, консистентная вёрстка; легко поддерживать дизайн-систему |
| Тесты | **Vitest + RTL + Playwright** (e2e, позже) | Быстрые юнит-тесты; e2e для критичных путей |
| i18n самого UI | **i18next** (опц.) | Сам сервис тоже стоит сделать многоязычным |

## Почему таблица — это TanStack Table, а не «обычная» таблица

Требование заказчика: «бери в расчёт лучшую производительность; если есть более удачное
решение — предложи». Анализ:

- **Обычный `<table>` с map по строкам** — деградирует на сотнях/тысячах строк ×
  десятках колонок: тысячи DOM-узлов, тормозящий скролл и ввод. **Не подходит** для
  нашего объёма данных.
- **TanStack Table (headless)** — даёт модель таблицы (колонки, сортировка, фильтры,
  закреплённые колонки), но **не навязывает разметку**. Это позволяет подключить
  виртуализацию и кастомные ячейки-редакторы. ✅ Базовый выбор.
- **TanStack Virtual поверх неё** — рендерит только видимые строки (и, при многих языках,
  виртуализирует колонки). Это и есть ключ к «самой стабильной работе». ✅
- **AG Grid** — мощная готовая грид-система с виртуализацией из коробки. Альтернатива,
  если нужны сразу богатые фичи (груп­пировка, Excel-экспорт), но тяжелее, частично
  платная и менее гибкая по стилю. **Запасной вариант**, если TanStack+Virtual окажется мало.

**Рекомендация:** **TanStack Table + TanStack Virtual**. Это самый гибкий и стабильный
путь под наш кейс (ключ × много языков, инлайн-редактирование, sticky-колонки). Детали и
бенч-ориентиры — в `08-table-design.md`.
