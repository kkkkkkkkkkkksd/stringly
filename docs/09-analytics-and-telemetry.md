# 09. Продуктовая аналитика и телеметрия

Помимо «аналитики по запросам» (data-слой, см. `07`) полезно с самого начала заложить
**масштабируемую продуктовую аналитику** — чтобы понимать, как используется сервис, и
расти на данных. Реализуется тонким слоем, не зависящим от конкретного провайдера.

## Принципы

1. **Провайдеро-независимый интерфейс** `analytics.track(event, props)` — за ним может
   стоять что угодно (свой бэк, PostHog, Amplitude, заглушка-логгер).
2. **Типизированные события** — список событий описан как union-тип, нельзя отправить
   «опечатку».
3. **Без PII по умолчанию** — не шлём email/тексты переводов; только идентификаторы и метрики.
4. **Батчинг и отказоустойчивость** — события буферизуются и отправляются пачками; сбой
   аналитики никогда не ломает UI.

## Интерфейс

```ts
type AnalyticsEvent =
  | { name: 'app_opened' }
  | { name: 'signed_up'; props: { method: 'email' } }
  | { name: 'logged_in' }
  | { name: 'project_switched'; props: { projectId: string } }
  | { name: 'namespace_viewed'; props: { namespace: string } }
  | { name: 'language_added'; props: { code: string } }
  | { name: 'key_added'; props: { namespace: string } }
  | { name: 'translations_saved'; props: { count: number } }
  | { name: 'token_created' }
  | { name: 'search_used'; props: { hasResults: boolean } };

interface Analytics {
  track(e: AnalyticsEvent): void;
  identify(userId: string): void;
  page(path: string): void;
}
```

Реализация по умолчанию (пока нет бэка) — `consoleAnalytics` (логирует) либо буфер,
который позже сольётся на эндпоинт `/api/v1/analytics/batch`.

## Что меряем (стартовый набор)

**Активация и воронка:** заходы на лендинг → регистрации → первый созданный ключ →
первый выпущенный токен (момент «продукт заработал»).

**Вовлечённость:** активные проекты, число правок переводов в неделю, частота
переключения разделов, использование поиска/фильтров.

**Здоровье локализации (доменные метрики, считаются на бэке из БД):**
- % заполненности по каждому языку и разделу;
- число пустых ключей;
- «свежесть» — давность последнего изменения.
Эти метрики показываются на дашборде проекта (см. `02-features.md`).

**Технические:** время загрузки страницы строк, частота ошибок API, попадание в кэш.

## Масштабируемость аналитики

- **Слой абстракции** → смена провайдера = одна реализация интерфейса, без правок в фичах.
- **Серверная агрегация:** сырые события льются в БД/хранилище, доменные метрики
  («% заполненности») считаются на бэке и кэшируются — фронт получает готовые числа.
- **Батч-эндпоинт** `POST /analytics/batch` принимает массив событий → меньше запросов.
- **Семплирование** для высокочастотных событий (скролл, ввод), если объём вырастет.
- **Привязка к версии** клиента (`appVersion`) для анализа регрессий.

## Приватность

- Не логируем содержимое переводов и email в свойствах событий.
- Идентификация по `userId` (UUID), а не по email.
- Учитываем согласие пользователя (можно отключить трекинг в настройках) — пункт на v2.
