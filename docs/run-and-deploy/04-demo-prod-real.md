# Конфигурация 4 — Демо прод (реальный бэк на сервере)

Боевая публикация: фронт на сервере ходит в **реальный Kotlin-бэк**. Моки выключены. Это
финальный режим, когда бэк готов и крутится на сервере.

## Когда использовать

- Бэк реализован и запущен на сервере.
- Нужен настоящий многопользовательский доступ (регистрация/вход/проекты/участники).

## Предусловия на сервере

1. **Запущен Kotlin-бэк** на сервере, слушает локально (рекомендуется `127.0.0.1:8080`).
   Держать его лучше под systemd/Docker, чтобы не падал и автозапускался.
2. **nginx проксирует `/api/v1` на бэк.** В `/etc/nginx/sites-available/stringly`
   раскомментируй блок `location /api/v1/ { proxy_pass http://127.0.0.1:8080; ... }`
   (он заготовлен в `deploy/nginx-stringly.conf`) и перезагрузи nginx:
   ```bash
   ssh root@144.31.235.244 'nginx -t && systemctl reload nginx'
   ```

## Команда (каждый деплой)

С твоей машины, из корня проекта:

```bash
./deploy/deploy.sh real
```

Открывать: **https://144.31.235.244.nip.io:8443**.

## Что происходит

- `deploy.sh real` собирает фронт в режиме `demo-real` (`.env.demo-real`, `VITE_API_MODE=real`)
  → без MSW.
- `dist/` заливается на сервер, nginx раздаёт статику.
- Фронт шлёт запросы на `/api/v1/...`; nginx проксирует их на бэк `127.0.0.1:8080`. Один origin
  → без проблем с CORS.

## Нюансы

- HTTPS здесь не только из-за моков (их нет), а просто как норма для боевого доступа: вход и
  токены не должны ходить по голому HTTP. Самоподписанный сертификат работает, но к этому
  моменту лучше уже завести домен + Let's Encrypt (см. `05-server-setup.md`).
- Бэк отвечает по путям `/api/v1/...` (контракт — в `docs/backend`).
- Если на сервере одновременно нужны демо-на-моках и боевой — это два разных деплоя (mock/real);
  держать их одновременно на одном адресе нельзя, переключаешься пересборкой.
