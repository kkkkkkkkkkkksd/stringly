# Конфигурация 2 — Локал прод (реальный бэк)

Локальный запуск прод-сборки против **реального бэкенда**. Моки выключены. Нужно, чтобы
проверить интеграцию фронта с настоящим API до публикации.

## Когда использовать

- Бэк уже запущен (локально или доступен по сети) и хочешь проверить связку «как в проде».
- Отладка реальных ответов/ошибок API, которых нет в моках.

## Предусловие: запущенный бэк

Подними Kotlin-бэк локально (по умолчанию ожидается `http://localhost:8080`). Если бэк на
другом адресе — задай его через `BACKEND_ORIGIN` (см. ниже).

## Команда

```bash
npm run start:local-real
```

Это соберёт прод-версию в режиме `real` и поднимет `vite preview` на `http://localhost:4173`.

Если бэк не на `localhost:8080`:

```bash
BACKEND_ORIGIN=http://localhost:9000 npm run start:local-real
```

## Что происходит

- Режим сборки `local-real` → `.env.local-real` (`VITE_API_MODE=real`). MSW не стартует.
- Фронт шлёт запросы на относительный `/api/v1/...`.
- `vite preview` проксирует `/api/v1` → `BACKEND_ORIGIN` (по умолчанию `http://localhost:8080`),
  см. `vite.config.ts`. Прокси убирает проблему CORS — фронт и бэк выглядят одним origin.

## Нюансы

- Это **сборка**, а не dev-сервер: горячей перезагрузки нет (для итераций по UI используй
  конфиг 1). Цель здесь — проверить прод-поведение с реальными данными.
- Если хочется dev-сервер против бэка (с HMR), можно вместо этого запустить
  `BACKEND_ORIGIN=... vite --mode local-real` — прокси работает и в dev.
- Бэк должен отвечать по путям `/api/v1/...` (см. `docs/backend`).
