# 11. Роли и права доступа

Доступ — на уровне проекта (per-project). Один пользователь может иметь разные роли в
разных проектах. Роль хранится в `Membership` (см. `06-data-model.md`).

## Роли

| Роль | Кто это | Назначение |
|------|---------|-----------|
| **admin** | Владелец, создавший проект | Полный контроль над проектом |
| **translator** | Переводчик/редактор | Работа с переводами |
| **viewer** | Наблюдатель | Только просмотр |

В MVP создатель проекта получает роль `admin` автоматически. Назначение ролей другим —
v2 (см. F-27); до этого в проекте по сути один admin.

## Матрица прав

| Действие | admin | translator | viewer |
|----------|:-----:|:----------:|:------:|
| Просмотр таблицы переводов | ✅ | ✅ | ✅ |
| Просмотр списка участников | ✅ | ✅ | ✅ |
| Редактирование ячеек / сохранение | ✅ | ✅ | — |
| Добавление ключей | ✅ | ✅ | — |
| Добавление/удаление языков | ✅ | — | — |
| Управление участниками и ролями | ✅ | — | — |
| Выпуск/отзыв API-токенов | ✅ | — | — |
| Создание проектов | ✅ | — | — |
| Удаление проекта | ✅ | — | — |

> Список участников **виден всем** ролям (read-only), но изменять состав и роли может
> только admin — поэтому в UI кнопки редактирования просто скрыты/задизейблены для
> остальных.

## Применение на фронте

Права проверяются на двух уровнях (UI — для удобства, бэк — для безопасности):

```ts
// shared/lib/permissions.ts
const CAN: Record<Role, Set<Permission>> = {
  admin:      new Set(['table:read','table:write','keys:add','lang:manage',
                        'members:manage','tokens:manage','project:manage']),
  translator: new Set(['table:read','table:write','keys:add']),
  viewer:     new Set(['table:read']),
};

export const can = (role: Role, p: Permission) => CAN[role].has(p);
```

Использование:

- **Хук** `usePermission('members:manage')` → булево от роли в активном проекте.
- **Гард-компонент** `<Can perm="lang:manage">…</Can>` прячет управляющие элементы.
- **Роут-гард** `RequireRole` для admin-only страниц (если понадобятся).

Это не заменяет проверку на бэке: фронтовые ограничения — UX, реальная защита — на сервере
(каждый мутационный эндпоинт проверяет роль по токену сессии).
