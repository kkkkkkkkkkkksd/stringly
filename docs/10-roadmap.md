# 10. План работ (Roadmap)

Поэтапный план. Каждый этап даёт работающий результат. До появления бэка вся работа идёт
на mock-слое (MSW), поэтому фронт можно довести до полностью кликабельного состояния.

## Этап 0 — Фундамент (1–2 дня)

- Инициализация: Vite + React + TypeScript (strict).
- ESLint + Prettier + `eslint-plugin-boundaries` (правила слоёв из `04`).
- Структура папок (FSD-lite).
- Базовые провайдеры: `QueryClientProvider`, роутер, тема/Tailwind.
- `shared/config` с брендом (название **Stringly**) и env (`VITE_API_MODE`).
- MSW: каркас + первые фикстуры из таблицы-референса.

**Готово, когда:** приложение запускается, пустой роутинг работает, MSW перехватывает запросы.

## Этап 1 — Аутентификация + первый проект (3–4 дня)

- Экраны: лендинг, `/login`, `/register` (React Hook Form + Zod).
- HTTP-клиент с интерсептором (Bearer; на 401 — разлогин). **Без refresh** — токен бессрочный.
- `features/auth`: api, хуки (`useLogin`, `useRegister`, `useMe`), Zustand-стор сессии.
- `ProtectedRoute` + `RequireProject` (0 проектов → `/onboarding`).
- **Онбординг:** `/onboarding` — создание первого проекта (имя) → активный проект.

**Готово, когда:** можно зарегистрироваться/войти, создать первый проект и попасть в `/app`.

## Этап 2 — Каркас приложения (1–2 дня)

- Layout `/app`: левая панель (Таблица, Настройки), «Выйти».
- **Переключатель проектов** и **«+ Проект»** — UI-**заглушки** (показывают активный
  проект, переключение/создание неактивны). Активный проект в Zustand + localStorage,
  завязан на `pid` во всех query keys — мультипроектность включится позже без переделок.
- Слой прав: `shared/lib/permissions`, хук `usePermission`, компонент `<Can>` (см. `11`).
- Nested routes для настроек, заглушки страниц.

**Готово, когда:** навигация работает, контекст данных привязан к активному проекту.

## Этап 3 — Таблица переводов, чтение (3–5 дней) ⭐

- `features/namespaces`: вкладки разделов + создание раздела с выбором **типа**
  (`strings` / `plurals`).
- `features/languages`: централизованный список языков проекта → источник колонок.
- `features/translations-table`: TanStack Table + TanStack Virtual.
- Sticky-колонки `code`/`comment`, языковые колонки (из языков проекта), RTL.
- Серверная пагинация/инфинит-скролл. Поиск и фильтры — **UI-заглушка** (задизейблены),
  реализация в следующей итерации.
- Skeleton/пустые состояния.

**Готово, когда:** таблица рисует данные из mock и плавно скроллит большой объём.

## Этап 4 — Таблица: редактирование и сохранение (3–4 дня) ⭐

- Инлайн-редактор ячеек, буфер dirty в Zustand, точечные подписки, `React.memo`.
- Индикация изменений, счётчик на «Сохранить», guard на уход со страницы.
- Батч-`PATCH` с оптимистичным апдейтом и откатом.
- **+ Ключ** и **+ Язык** (модалы + мутации). **+ Язык** — быстрый доступ прямо из
  таблицы (тот же централизованный список, что в настройках).

**Готово, когда:** можно править, добавлять ключи/языки (в т.ч. из таблицы) и сохранять пачкой.

## Этап 5 — Настройки: языки, участники (2–3 дня)

- **Языки** `/app/settings/languages` (дефолтная вкладка): централизованное
  добавление/удаление, базовый язык, RTL. Добавление → колонка во всех разделах
  (инвалидация `qk.languages`).
- **Участники** `/app/settings/members`: список с ролями, виден всем; управление —
  только admin (через `<Can perm="members:manage">`). Приглашение может быть заглушкой (v2).
- **Предпочтения** `/app/settings/preferences` — тема light/dark/system (**готово**). Язык
  интерфейса приложения — **пока только русский** (i18n самого Stringly — later).
- **Подключения** `/app/settings/connections` — вкладка-**заглушка** («скоро»); API-токены реализуем позже.
- **Документация** `/app/settings/documentation` — вкладка-**заглушка** (наполнение позже).
- **История изменений** `/app/settings/history` (F-17) — **обязательная фича**, в MVP
  вкладка-**заглушка** (кнопка/раздел есть, наполнение — v2: кто/что/когда + фильтры).
- Заглушки профиля/проекта (v2): Профиль — смена email/пароля + инфо аккаунта; Проект —
  переименование, базовый язык, slug, формат экспорта, удаление.

**Готово, когда:** язык добавляется централизованно (из настроек и из таблицы), виден список участников с ролями.

## Этап 6 — Аналитика и полировка (2–3 дня)

- Слой `analytics` + типизированные события (`09`).
- Дашборд-заглушка: % заполненности по языкам (из mock-агрегата).
- Тосты ошибок, Error Boundary, доступность (клавиатура в таблице).
- Юнит-тесты ключевой логики (буфер правок, query keys, валидаторы).

**Готово, когда:** базовая телеметрия шлётся, критичные пути покрыты тестами.

> **Локальная разработка:** этапы 0–6 целиком ведутся локально на `http://localhost:5173`
> (Vite dev-сервер). Реальный домен/хост/URL не нужен — он покупается и подключается на
> Этапе 7. Сайт открывается в браузере локально, данные отдаёт MSW.

## Этап 7 — Интеграция с реальным бэком + домен (в самом конце)

- Реализовать контракт из `07` на бэке (та же модель данных из `06`).
- Переключить `VITE_API_MODE=real`, выключить MSW.
- Купить домен, настроить хостинг/деплой, прод-сборка (`npm run build`).
- e2e-тесты (Playwright) на критичные сценарии.
- При необходимости — пересмотр пагинации под реальные объёмы.

## Бэклог будущих релизов (после MVP)

Этапы 0–7 закрывают MVP. Дальше — итерации поверх готового фундамента (данные уже через
React Query + `qk`, статус-поле и `pid` заложены, так что фичи лягут без переделок):

- **Статусы перевода — workflow (F-29).** Per-cell статус как в Xcode String Catalog:
  `new` (ключ добавлен, перевода нет — нужно внимание переводчика), `needs review`
  (изменился исходный текст/ключ → перевод устарел), `translated` (актуально).
  Авто-простановка `needs review` при изменении базового значения; индикация в ячейке
  (пилюля из `14-design-system`, статусы уже описаны там) + фильтр по статусу (вместе с F-14).
  Базируется на существующем `Translation.status` (06).
- **Версионирование / релизные каналы (F-30, F-31).** Типичный кейс: что-то в релизе,
  что-то на тесте, что-то только планируется. Каналы `planned → testing → released`,
  продвижение между ними, снимок (snapshot) релиза по версии/тегу, diff между версиями,
  откат. Публичный API отдаёт перевод для выбранного канала (`?channel=released`).
  Дополняет аудит F-17 (история «кто/когда»), не заменяет его.
- **Импорт/экспорт (F-16), история/аудит (F-17), автоперевод (F-19)** — по приоритету.

Порядок предложений: сначала **статусы** (дешёвый слой поверх таблицы, заметная польза
переводчикам), затем **версионирование** (более крупная фича — затрагивает модель данных,
публичный API и UI выбора канала).

## Definition of Done для MVP

Регистрация/вход работают (один бессрочный токен, без refresh); при первом входе
создаётся проект; переключатель проектов и «+ Проект» — заглушки, данные привязаны к
активному проекту; таблица читает и плавно скроллит большой объём (поиск/фильтр —
неактивный UI); разделы создаются с выбором типа (strings/plurals); ячейки редактируются
и сохраняются батчем; добавляются ключи; **языки добавляются централизованно** (из
настроек и из таблицы) и появляются во всех разделах; виден список участников с ролями;
вкладка «Подключения» — заглушка; всё переключается на реальный бэк сменой одного флага.
