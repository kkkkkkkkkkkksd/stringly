# 13. Дизайн и UX (design-first workflow)

Дизайн-система с нуля разрабатывается вместе с агентами. Чтобы не переделывать экраны,
действует правило **design-first**: перед версткой любого экрана/флоу агент сначала
**предлагает варианты дизайна на выбор**, и только после утверждения пишет код.

## Правило design-first

Для каждого нового экрана или флоу:

1. **Предложить 2–3 варианта** дизайна (разные по компоновке/акцентам), кратко описать
   плюсы/минусы каждого. Визуализировать как лёгкий прототип (HTML/JSX-мокап или
   схема-вайрфрейм), не как финальный код.
2. **Пользователь выбирает** вариант (или комбинацию) и даёт правки.
3. Только потом — **реализация** утверждённого варианта в коде по архитектуре проекта.

Это экономит время: дешевле перебрать макеты, чем переписывать готовые компоненты.

## Дизайн-система (ЗАФИКСИРОВАНА)

Визуал утверждён и вынесен в **`14-design-system.md`** — это источник правды по цветам,
шрифтам и токенам. Кратко:

- **Направление:** A · Indigo & Slate. **Шрифт:** Inter (+ JetBrains Mono для ключей).
- **Темы:** светлая и тёмная (через CSS-переменные).
- **Токены:** палитра, типографика, отступы, радиусы, тени — в `14` (готовые CSS-переменные
  и Tailwind-конфиг).
- **Базовые компоненты (`shared/ui`):** Button, Input, Select, Modal, Toast, Table-обёртка,
  Tabs, Badge/StatusPill, EmptyState, Skeleton — спецификация в `14`, §5.

На дизайн-шаге не придумывай новые цвета/размеры — собирай из токенов `14`. Нужен новый
токен — добавь его в `14`, не плоди разовые значения. Все экраны — из общих компонентов,
отдельные «снежинки» не плодим.

## Принципы UX продукта

- Рабочий инструмент → плотность информации важнее «воздуха», но без визуального шума.
- Таблица — главный экран: читаемость ячеек, явная индикация пустых/dirty/статусов,
  понятный sticky-заголовок и закреплённые колонки.
- Действия, меняющие данные, дают мгновенный отклик (оптимистично) и понятную индикацию.
- Состояния не забываем: loading (skeleton), empty, error, no-permission (read-only для viewer).
- Доступность: контраст, фокус с клавиатуры, навигация по таблице стрелками, поддержка RTL.

## Что предлагать для ключевых экранов

| Экран / флоу | На что давать варианты |
|--------------|------------------------|
| Лендинг | структура hero, расположение CTA, тон |
| Аутентификация | одна страница с табами vs отдельные login/register; центр vs сплит |
| Каркас /app | ширина/вид сайдбара, расположение переключателя проекта и профиля |
| Таблица | вид вкладок-разделов, плотность строк, оформление dirty/статусов, тулбар |
| Редактор ячейки | инлайн vs поповер; как показывать plural-формы |
| Настройки | вертикальные вкладки vs горизонтальные; вид списка участников/языков |

## Чек-лист дизайн-шага

- [ ] Предложены 2–3 варианта, у каждого — плюсы/минусы.
- [ ] Использованы токены и компоненты дизайн-системы (или предложено дополнение к ней).
- [ ] Учтены состояния: loading / empty / error / read-only (роль viewer).
- [ ] Учтены адаптив и RTL (для таблицы — закреплённые колонки и скролл).
- [ ] Вариант утверждён пользователем до начала реализации.

## Инструменты

- Прототипы — лёгкие мокапы (HTML/JSX или вайрфрейм-схемы), быстрые к правкам.
- Реализация — Tailwind + компоненты `shared/ui`, токены в конфиге темы.
- Финальная сверка свёрстанного экрана со утверждённым макетом (скриншот-ревью).
