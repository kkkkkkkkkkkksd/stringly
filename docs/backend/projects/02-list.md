# 05. Список проектов пользователя — `GET api/v1/projects`

Возвращает все проекты, где текущий пользователь участник: и свои (где он admin), и чужие,
куда его пригласили. У каждого проекта — роль текущего пользователя. Это и есть «один
пользователь видит проекты разных компаний» из аналогии с Google Таблицами.

**Закрывает:** пункты ТЗ 7 и 8 (мультидоступ, проекты разных владельцев в одном списке).

## Эндпоинт

| | |
|---|---|
| Метод | `GET` |
| URL | `api/v1/projects` |
| Авторизация | обязательна — `Bearer` |

## Query-параметры

| Параметр | Тип | По умолч. | Назначение |
|----------|-----|-----------|------------|
| `limit` | number | 50 | размер страницы (1–100) |
| `cursor` | string | — | курсор следующей страницы |

## Ответы

### 200 OK

```json
{
  "items": [
    {
      "id": "prj_018f...aaa",
      "name": "Acme Mobile App",
      "slug": "acme-mobile-app",
      "role": "admin",
      "createdAt": "2026-06-11T09:05:00Z"
    },
    {
      "id": "prj_018f...bbb",
      "name": "Globex Web",
      "slug": "globex-web",
      "role": "translator",
      "createdAt": "2026-05-20T12:00:00Z"
    }
  ],
  "nextCursor": null
}
```

> Фронт сейчас ждёт «голый» массив (`z.array(projectSchema)`). Рекомендуется привести фронт к
> конверту `{ items, nextCursor }` ради единообразной пагинации (см. README). На первое время
> бэк может отдавать массив, но целевой контракт — конверт.

### Ошибки

| HTTP | code | Причина |
|------|------|---------|
| 401 | `UNAUTHENTICATED` | нет токена |

## Бэкенд: логика

1. Аутентификация → `currentUser`.
2. `SELECT p.*, m.role FROM memberships m JOIN projects p ON p.id = m.project_id
   WHERE m.user_id = currentUser.id ORDER BY p.created_at DESC, p.id` с keyset-пагинацией по
   `(created_at, id)` и `limit+1` для вычисления `nextCursor`.
3. Замапить в `{ id, name, slug, role, createdAt }`. `role` — из membership текущего пользователя.

Производительность: индекс `memberships(user_id)` + join по PK проекта. Масштабируется на
тысячи проектов/пользователя за счёт keyset-пагинации.

## Фронтенд: логика

- **Слой:** `features/projects`.
- **API:** `projectsApi.list()` → `httpClient.get('/projects')` → парс схемой коллекции.
- **Хук:** `useProjects()` — `useQuery({ queryKey: qk.projects, queryFn: projectsApi.list })` (есть).
- **Применение:** `ProjectSwitcher` (переключатель), гард `RequireProject`, роль активного
  проекта через `useCurrentProject()/useRole()` (есть). При мультидоступе свитчер показывает
  проекты всех «компаний» вперемешку — можно позже сгруппировать по владельцу.
- **Состояния:** loading (скелетон свитчера), empty (нет проектов → редирект в онбординг).

## Edge cases

- Пустой список → пользователь только что зарегистрировался → онбординг (создать проект).
- Группировка по владельцу/организации — точка расширения (когда появятся организации).
