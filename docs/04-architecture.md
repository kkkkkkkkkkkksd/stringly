# 04. Архитектура фронтенда

Цель — **масштабируемость и понятность**: новый разработчик быстро находит, где что
лежит; фичи добавляются без переписывания; бизнес-логика отделена от UI и от транспорта.

## Принципы

1. **Feature-Sliced** подход (упрощённый FSD). Код растёт по фичам, а не по типам файлов.
2. **Разделение слоёв:** UI ↔ бизнес-логика (hooks) ↔ слой данных (api + React Query) ↔ транспорт (HTTP-клиент).
3. **Серверное состояние** живёт в React Query, **клиентское** (UI/буфер правок) — в Zustand. Не смешиваем.
4. **Типобезопасность** end-to-end: TypeScript + Zod-схемы на границе API.
5. Бэка пока нет → весь доступ к данным идёт через **mock-адаптер** за единым интерфейсом,
   который позже без изменений в фичах заменяется на реальный HTTP (см. `07-api-and-queries.md`).

## Структура папок

```
src/
├── app/                      # инициализация приложения
│   ├── providers/            # QueryClientProvider, RouterProvider, тема
│   ├── router.tsx            # дерево маршрутов + гарды
│   └── App.tsx
│
├── pages/                    # экраны = композиция фич (тонкий слой)
│   ├── landing/
│   ├── auth/                 # login, register
│   ├── table/
│   └── settings/             # connections, profile, project
│
├── features/                 # самостоятельные функциональные блоки
│   ├── auth/                 # login/register/refresh: api, hooks, store, ui
│   ├── translations-table/   # ядро: таблица, ячейки, редактирование
│   ├── languages/            # добавление/список языков
│   ├── namespaces/           # вкладки-разделы
│   └── api-tokens/           # выпуск/отзыв токенов
│
├── entities/                 # доменные сущности + их типы и базовые UI
│   ├── project/
│   ├── translation/
│   ├── language/
│   └── token/
│
├── shared/                   # переиспользуемое, без доменной логики
│   ├── services/             # СЕРВИСЫ: интеграции/сайд-эффекты (network: httpClient;
│   │   │                     #   далее analytics, storage). Аналог Services-таргета в iOS.
│   │   └── network/          #   сетевой слой: httpClient, authToken
│   ├── core/                 # CORE: чистые хелперы/правила без сайд-эффектов
│   │   │                     #   (validation, format…). Аналог Core-таргета в iOS.
│   │   ├── validation.ts
│   │   └── format.ts
│   ├── api/                  # ключи кэша React Query (qk-фабрика)
│   ├── ui/                   # библиотека компонентов (баррель @/shared/ui)
│   ├── resources/            # РЕСУРСЫ: ассеты + тексты
│   │   ├── assets/           #   иконки/изображения (именованные компоненты)
│   │   └── i18n/             #   тексты UI (баррель @/shared/resources/i18n)
│   ├── config/               # env, бренд (название Stringly), константы
│   └── types/
│
└── main.tsx
```

### Слои `services` и `core` (терминология iOS)

- **`shared/services`** — «сервисы»: всё, что общается с внешним миром или имеет
  сайд-эффекты. Сейчас это `network` (HTTP-клиент). Сюда же пойдут `analytics`, `storage`
  и подобные интеграции. Фичи зовут сервисы, а не делают «голый» fetch.
- **`shared/core`** — «ядро»: чистые переиспользуемые функции без сайд-эффектов —
  валидаторы, форматтеры, утилиты. Не знает о сети, UI и React.

Оба живут в `shared` (самый нижний слой), поэтому доступны откуда угодно «вниз». Правило:
сетевую/интеграционную логику выноси в `services`, чистые хелперы/валидаторы — в `core`.

### Правила зависимостей (важно для масштабируемости)

Слои импортируют только «вниз»:

```
app → pages → features → entities → shared
```

- `shared` не знает ни о чём выше.
- `features` не импортируют друг друга напрямую — общая логика выносится в `entities`/`shared`.
- `pages` только композируют фичи, своей логики почти не содержат.

Это правило можно форсить линтером (`eslint-plugin-boundaries`), чтобы архитектура не
размывалась со временем.

## Слой данных (коротко)

```
UI компонент
   │ вызывает
useTranslations() ──── кастомный hook (features/translations-table/model)
   │ под капотом
useQuery / useMutation (React Query)
   │ дёргает
translationsApi.getList()  ──── интерфейс (features/.../api)
   │ реализация
httpClient (shared/api)  ←→  mockAdapter (пока нет бэка)
```

Фичи зависят от **интерфейса** api, а не от конкретной реализации → переключение
mock → реальный бэк = смена одной фабрики в `shared/api`.

## Состояние

| Тип состояния | Где живёт | Примеры |
|---------------|-----------|---------|
| Серверное (из БД) | React Query кэш | переводы, языки, токены |
| Клиентское глобальное | Zustand | текущий проект, auth-токены, буфер dirty-ячеек |
| Локальное UI | useState/useReducer | открыт ли модал, текст поиска |
| Состояние форм | React Hook Form | формы логина/регистрации, создание токена |

## Аутентификация (упрощённо на первое время)

- При логине/регистрации сервер отдаёт **один бессрочный access-токен** (без refresh).
- Хранится в Zustand + `localStorage` (с TODO про безопасность).
- HTTP-клиент добавляет `Authorization: Bearer`. На `401` — разлогин и редирект на `/login`.
- Refresh/ротацию и `httpOnly` cookie добавим позже — слой HTTP-клиента изолирован,
  поэтому включение refresh не затронет фичи.

## Сборка и инструменты

- **Vite** — dev-сервер и сборка (быстро, нативный TS/JSX).
- **TypeScript** в strict-режиме.
- **ESLint + Prettier**, `eslint-plugin-boundaries` для слоёв.
- **Vitest + React Testing Library** — юнит/компонентные тесты.
- **MSW (Mock Service Worker)** — мокаем сеть как настоящую (см. ниже).
- **IDE:** VS Code (рекомендованные расширения и настройки — в корневом `README.md`).

## Запуск и окружения (локальная разработка)

Весь MVP разрабатывается и проверяется **локально**. Реальный домен/хост/URL не нужен до
самого конца — он покупается и подключается на финальном этапе (см. `10-roadmap`, Этап 7).

- `npm run dev` поднимает сайт на `http://localhost:5173` (Vite). Этого достаточно, чтобы
  открыть приложение в браузере и вести всю разработку.
- Бэкенда нет → данные отдаёт MSW (мок-сеть) прямо в браузере; `VITE_API_MODE=mock`.
- Прод-домен, деплой и `VITE_API_MODE=real` — только когда появятся бэк и купленный домен.
- Никаких хардкод-URL в коде: базовый адрес API берётся из env (`VITE_API_BASE_URL`),
  по умолчанию относительный путь `/api`, чтобы локально и на проде работало без правок.
