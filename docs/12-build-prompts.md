# 12. Последовательность запросов к AI-агенту (build playbook)

Готовые промпты, чтобы строить проект **по шагам**. Запросы идут в порядке роадмапа
(`10-roadmap.md`). Каждый шаг самодостаточен, но опирается на результат предыдущего.

Главная идея: на **каждом** шаге агент сначала читает документацию и **смотрит вперёд** —
что ему предстоит делать дальше — и проектирует текущий код так, чтобы будущие шаги
ложились сверху без переделок.

> **Командный режим:** короткие фразы пользователя («засетапь проект», «сделай таблицу» …)
> сопоставлены с этими Шагами в `CLAUDE.md` → «Карта команд». Агент по фразе сам находит
> нужный Шаг здесь, читает указанные docs и начинает — длинный промпт каждый раз не нужен.
> Развёрнутые тексты ниже остаются как эталон/детализация шага.

---

## Как пользоваться

1. Давай запросы агенту **по одному**, по порядку. Не переходи к следующему, пока текущий
   не доведён до «Готово, когда».
2. В начале **каждого** запроса вставляй блок **«Правила агента»** (ниже) — это держит
   агента в рамках архитектуры и заставляет учитывать будущее.
3. **Дизайн-first:** для шагов с UI (1, 2, 3, 4, 5, лендинг) сначала запусти
   **дизайн-подшаг** — агент предлагает 2–3 варианта макета, ты выбираешь, и только потом
   идёт реализация. Шаблон дизайн-запроса — ниже, подробности в `13-design-and-ux.md`.
4. После шага бегло проверяй результат по чек-листу шага.

### Шаблон дизайн-запроса (перед каждым UI-шагом)

```
[Правила агента]
Прежде чем писать код экрана «<название>», предложи 2–3 варианта дизайна по
docs/13-design-and-ux: разные компоновки, кратко плюсы/минусы каждого, в виде лёгких
мокапов (HTML/JSX или вайрфрейм). Используй/предложи токены и компоненты дизайн-системы
(shared/ui). Учти состояния loading/empty/error/read-only и RTL. НЕ пиши финальный код,
жди выбора варианта.
```
На первом UI-шаге дополнительно попроси завести минимальную дизайн-систему (токены +
базовые компоненты), чтобы следующие экраны были консистентны.

## Правила агента (вставлять в каждый запрос)

```
ПРАВИЛА (соблюдай на каждом шаге):
1. Сначала прочитай docs/: README, 04-architecture, 05-tech-stack, 06-data-model,
   07-api-and-queries, и файл, относящийся к текущему шагу. Не противоречь им.
2. Посмотри ВПЕРЁД по docs/10-roadmap.md и docs/12-build-prompts.md: пойми, какие шаги
   будут после текущего, и спроектируй код так, чтобы они легли сверху БЕЗ переделок.
   Конкретно учитывай заранее:
   - всё привязано к активному проекту (pid) — даже если переключатель проектов сейчас
     заглушка, структура данных/ключей кэша должна это поддерживать;
   - доступ к данным идёт через интерфейс api + React Query (qk-фабрика ключей), а не
     прямой fetch в компонентах; реализация переключается mock↔real через VITE_API_MODE;
   - языки централизованы на уровне проекта (колонки таблицы берутся из них);
   - роли (admin/translator/viewer) и слой permissions — закладывай точки расширения,
     даже где сейчас один admin;
   - таблица будет виртуализирована и редактируема — не закладывай решений, ломающих
     производительность (см. 08-table-design).
3. Соблюдай структуру папок и правила зависимостей слоёв (app→pages→features→entities→shared).
4. TypeScript strict, Zod-схемы на границе API, без any.
5. Если решение влияет на будущие шаги — кратко опиши допущения и точки расширения в
   комментариях/README, не делай необратимых упрощений.
6. Не выходи за рамки текущего шага: то, что помечено «заглушка/later», оставляй заглушкой
   (видимый, но неактивный UI), но с готовой точкой подключения.
7. ДИЗАЙН-FIRST: для любого нового экрана/флоу сначала предложи 2–3 варианта макета и
   дождись выбора, и только потом верстай (см. docs/13-design-and-ux). Не приступай к
   финальному UI-коду без утверждённого варианта.
8. ПРОВЕРКА И ПЕРЕИСПОЛЬЗОВАНИЕ (в конце КАЖДОЙ задачи): сверься с тем, что сделано раньше
   (shared/ui, соседние фичи/экраны); любой стилизованный элемент, который повторяется или
   пригодится дальше, вынеси в src/shared/ui (баррель @/shared/ui) и переиспользуй. Не
   дублируй стили. Затем прогони typecheck/lint/test/build — зелёные. (См. CLAUDE.md →
   «После каждой задачи».)
9. ТЕКСТЫ: ни одной видимой строки в компонентах — все тексты берём из
   src/shared/resources/i18n (баррель @/shared/resources/i18n): общие в common, экранные в
   своём namespace. Сообщения валидации тоже оттуда. (См. docs/15-copy-and-i18n.)
10. СЕРВИСЫ И CORE: анализируй логику и выноси её. Сетевую/интеграционную логику и
    сайд-эффекты — в src/shared/services (network, analytics, storage…). Чистые
    переиспользуемые функции (валидаторы, форматтеры, утилиты) — в src/shared/core. Не
    держи это в компонентах/фичах. (См. docs/04, раздел про services и core.)
11. АССЕТЫ: иконки/изображения — в src/shared/resources/assets именованными компонентами
    (без «сырых» SVG-path в JSX). resources = assets + i18n.
```

## Шаг 0 — Каркас проекта

**Цель:** рабочий скелет (Vite + React + TS) со структурой папок, провайдерами и моками.

```
[Правила агента]

Засетапь проект по docs/04-architecture и docs/05-tech-stack:
- Vite + React 18 + TypeScript (strict), ESLint + Prettier, eslint-plugin-boundaries.
- Структуру папок из 04-architecture (app/pages/features/entities/shared).
- Провайдеры: QueryClientProvider, RouterProvider, тема/Tailwind.
- shared/config: бренд (название Stringly) и env-флаг VITE_API_MODE = mock|real.
- shared/api: httpClient (Bearer), пустая qk-фабрика ключей кэша.
- MSW: каркас + 1-2 заглушечных хендлера, чтобы доказать, что мок-сеть работает.
Не реализуй фичи — только фундамент. Поясни в README репозитория, как запускать.
```

**Готово, когда:** `npm run dev` поднимает приложение, линт слоёв работает, MSW перехватывает запрос.

## Шаг 1 — Аутентификация + первый проект

**Цель:** регистрация/вход (бессрочный токен), онбординг с созданием проекта.

```
[Правила агента]

Сделай аутентификацию по docs/03-screens (экраны 1-3.5) и docs/07-api-and-queries (Auth):
- Лендинг (/), /login, /register на React Hook Form + Zod.
- features/auth: api + хуки useLogin/useRegister/useMe, Zustand-стор сессии.
- ВАЖНО: один бессрочный access-токен, БЕЗ refresh. На 401 — разлогин.
- ProtectedRoute + RequireProject (0 проектов → /onboarding).
- /onboarding: создание первого проекта (только имя) → активный проект.
- MSW-хендлеры под контракт Auth и POST /projects.
Заложи на будущее: активный проект хранится так, чтобы переключатель (след. шаг) включался
без переделок; токен-слой изолирован, чтобы позже добавить refresh.
```

**Готово, когда:** можно зарегистрироваться/войти, создать проект, попасть в /app.

## Шаг 2 — Каркас приложения (переключатель — заглушка)

**Цель:** layout `/app`, левая панель, привязка контекста к активному проекту, слой прав.

```
[Правила агента]

Сделай каркас /app по docs/03-screens (экран 4) и docs/11-roles-and-permissions:
- Layout с левой панелью (Таблица, Настройки), «Выйти».
- Переключатель проектов и «+ Проект» — UI-ЗАГЛУШКИ (показывают активный проект,
  переключение/создание неактивны), но активный pid реально пробрасывается во все
  будущие query keys.
- shared/lib/permissions: can(role, perm), хук usePermission, компонент <Can>.
- Nested routes настроек со страницами-заглушками.
Заложи на будущее: чтобы включить мультипроектность позже, достаточно «оживить»
переключатель — данные уже завязаны на pid.
```

**Готово, когда:** навигация по /app работает, контекст привязан к проекту, <Can> готов.

## Шаг 3 — Таблица переводов (чтение)

**Цель:** разделы с типом, колонки из языков проекта, виртуализированный рендер.

```
[Правила агента]

Сделай таблицу (чтение) по docs/03-screens (экран 5), docs/08-table-design,
docs/06-data-model, docs/07-api-and-queries:
- features/namespaces: вкладки + создание раздела с выбором типа (strings | plurals).
- features/languages: централизованный список языков проекта → источник колонок таблицы.
- features/translations-table: TanStack Table + TanStack Virtual, sticky-колонки
  code/comment, языковые колонки из языков проекта, RTL для ar/he.
- Серверная пагинация/инфинит-скролл, skeleton/пустые состояния.
- Поиск и фильтры — ВИДИМЫЙ, НО НЕАКТИВНЫЙ UI (задизейблено); параметры заложи в api на будущее.
- MSW: засей данными из таблицы-референса (см. скриншот/ключи [copy], [try_again]…).
Заложи на будущее (шаг 4): ячейки будут редактируемыми и mемоизированными — рендер строки
не должен зависеть от глобального состояния так, чтобы потом мешать точечным правкам.
```

**Готово, когда:** таблица рисует mock-данные и плавно скроллит большой объём.

## Шаг 4 — Таблица: редактирование и сохранение

**Цель:** инлайн-правки, буфер dirty, батч-сохранение, добавление ключей/языков.

```
[Правила агента]

Добавь редактирование в таблицу по docs/08-table-design и docs/07-api-and-queries:
- Инлайн-редактор ячеек; буфер dirty-ячеек в Zustand с ТОЧЕЧНЫМИ подписками; React.memo
  на ячейках; стабильные columns/data (useMemo).
- Индикация dirty, счётчик на кнопке «Сохранить», guard на уход со страницы.
- Батч-PATCH с оптимистичным апдейтом и откатом (onMutate/onError/onSettled).
- «+ Ключ» (модал) и «+ Язык» (быстрый доступ из таблицы; тот же централизованный список,
  что в настройках — добавленный язык = колонка во всех разделах).
Для plural-разделов учти отдельный вид ячейки (формы CLDR) хотя бы на уровне типа.
```

**Готово, когда:** можно править, добавлять ключи/языки и сохранять пачкой; UI не лагает.

## Шаг 5 — Настройки: языки и участники

**Цель:** централизованное управление языками, список участников с ролями.

```
[Правила агента]

Сделай настройки по docs/03-screens (раздел 6) и docs/11-roles-and-permissions:
- /app/settings/languages (дефолт): добавление/удаление, базовый язык, RTL. Это ТОТ ЖЕ
  список, что и быстрый доступ из таблицы (не дублируй логику — переиспользуй features/languages).
- /app/settings/members: список участников с ролями, виден всем (read-only для не-admin),
  управление через <Can perm="members:manage"> (в MVP admin один; приглашение — заглушка).
- /app/settings/connections — вкладка-ЗАГЛУШКА («скоро»); API-токены реализуем позже,
  но оставь точку подключения по контракту из 07.
```

**Готово, когда:** язык добавляется централизованно (из настроек и из таблицы), виден список участников с ролями.

## Шаг 6 — Аналитика и полировка

**Цель:** телеметрия, дашборд-заглушка, обработка ошибок, тесты.

```
[Правила агента]

Заверши MVP по docs/09-analytics-and-telemetry:
- Слой analytics (track/identify/page) + типизированные события; реализация по умолчанию —
  логгер/буфер, провайдеро-независимый интерфейс.
- Дашборд-заглушка: % заполненности по языкам из mock-агрегата.
- Глобальная обработка ошибок (тосты), Error Boundary на страницах, доступность таблицы
  (навигация с клавиатуры).
- Юнит-тесты (Vitest) на: буфер dirty-правок, qk-фабрику ключей, Zod-валидаторы, permissions.
```

**Готово, когда:** телеметрия шлётся, критичные пути покрыты тестами, UI отполирован.

## Шаг 7 — Интеграция с реальным бэком (когда появится)

```
[Правила агента]

Подключи реальный бэк по docs/07-api-and-queries и docs/06-data-model:
- Реализация api переключается на real (VITE_API_MODE=real), MSW выключается.
- Проверь соответствие ответов Zod-схемам; почини расхождения с контрактом.
- Включи ранее заглушённые фичи по мере готовности бэка: переключатель/создание проектов,
  API-токены, поиск/фильтры, refresh-токены.
- e2e-тесты (Playwright) на критичные сценарии: вход → таблица → правка → сохранение.
```

**Готово, когда:** приложение работает на боевом API без правок в слое фич.
